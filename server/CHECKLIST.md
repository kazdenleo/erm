# ✅ Чеклист проверки после подготовки проекта

## 1. Установка зависимостей
- [x] Запустить `npm install`
- [ ] Проверить, что все зависимости установлены без ошибок

## 2. Настройка окружения
- [ ] Создать `.env` файл из `.env.example`
- [ ] Настроить переменные окружения:
  - [ ] `NODE_ENV=development`
  - [ ] `PORT=3001`
  - [ ] `CLIENT_URL=http://localhost:3000`
  - [ ] `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`
  - [ ] `USE_POSTGRESQL=true` (если используется PostgreSQL)

## 3. Проверка конфигурации
- [ ] Проверить, что `src/config/index.js` загружается без ошибок
- [ ] Убедиться, что валидация Zod работает корректно
- [ ] Проверить, что все пути к файлам корректны

## 4. Проверка базы данных
- [ ] Убедиться, что PostgreSQL запущен (если используется)
- [ ] Проверить подключение: `npm run test:system`
- [ ] Проверить, что пул соединений инициализируется корректно

## 5. Запуск сервера
- [ ] Запустить сервер: `npm run dev`
- [ ] Проверить, что сервер запускается без ошибок
- [ ] Проверить логи в консоли

## 6. Проверка endpoints
- [ ] `GET /health` - должен вернуть статус сервера и БД
- [ ] `GET /api/products` - должен вернуть список товаров
- [ ] `GET /api/orders` - должен вернуть список заказов
- [ ] Проверить, что валидация работает (попробовать отправить невалидные данные)

## 7. Проверка логирования
- [ ] Проверить, что папка `logs/` создается автоматически
- [ ] Проверить, что логи записываются в файлы:
  - [ ] `logs/app-YYYY-MM-DD.log`
  - [ ] `logs/error-YYYY-MM-DD.log`
- [ ] Проверить формат логов (красивый в dev, JSON в prod)

## 8. Проверка безопасности
- [ ] Проверить, что Helmet работает (заголовки безопасности)
- [ ] Проверить CORS (должен разрешать только CLIENT_URL)
- [ ] Проверить rate limiting (попробовать отправить много запросов)
- [ ] Проверить ограничение размера JSON (попробовать отправить большой запрос)

## 9. Проверка валидации
- [ ] `POST /api/products` с невалидными данными - должна вернуться ошибка 400
- [ ] `PUT /api/products/:id` с невалидными данными - должна вернуться ошибка 400
- [ ] Проверить валидацию для других endpoints (warehouses, suppliers, orders)

## 10. Проверка обработки ошибок
- [ ] Проверить, что ошибки логируются в `logs/error-*.log`
- [ ] Проверить формат ответа об ошибке (должен быть структурированным JSON)
- [ ] Проверить, что stack trace показывается только в development

## 11. Запуск тестов
- [ ] Создать `.env.test` файл (если нужен)
- [ ] Запустить тесты: `npm test`
- [ ] Проверить, что все тесты проходят
- [ ] Проверить coverage отчет

## 12. Проверка graceful shutdown

## 14. Проверка middleware

## 🐛 Известные проблемы и решения

### Проблема: Jest не работает с ES modules
**Решение:** Используйте команду `npm test` (уже настроена с `--experimental-vm-modules`)

### Проблема: Валидация конфигурации падает
**Решение:** Убедитесь, что все обязательные переменные окружения установлены в `.env`

### Проблема: База данных не подключается
**Решение:** 
1. Проверьте, что PostgreSQL запущен
2. Проверьте настройки в `.env`
3. Проверьте логи: `logs/error-*.log`

### Проблема: Логи не создаются
**Решение:** Убедитесь, что у приложения есть права на запись в папку `logs/`
